빌더 패턴

25.10.2024 — java — 9 min read

(도입 - 흔한 실수 사례)

다음 주에 올라가야하는 feature에서 잘못된 Spec으로 개발되고 있다는 이야기를 들었다. Client to Server로 호출하고 있는 API가 있었는데 사실은 Server to Server로 호출이 되어야 하는 API 였던 것이다. 꽤나 경력을 가진 외주 분께 해당 API 호출을 Server to Server 방식으로 변경을 요청했다. 단순한 API 라 금방 끝날 줄 알았는데, 처음 컨트롤러에서 request를 받는 것 부터 문제를 겪고 계셨다.

🚨 Console 창에 나오는 Spring에서 자주 보는 그 에러
"com.fasterxml.jackson.databind.exc.InvalidDefinitionException: Cannot construct instance of ..."

문제는 @Builder 에 있었다. request 파라미터 역할을 하는 DTO 쪽에 클래스 스코프로 @Builder 가 생성되어 있었던 것이다. 코드를 같이 보며 @Builder 생성시에 생성자 때문에 직렬화 문제 생기는 거라고 설명을 드렸다. 그러면서 동시에 우리 팀 내에서 @Builder와 jackson의 직렬화/역직렬화 부분에 대해 충분히 설명을 해야겠다는 생각이 들었다.

Jackson과 @Builder의 관계

먼저 이해를 해야하는 것은 Spring에서 MessageConverter가 JSON을 직렬화/역직렬화 한다는 것이다. (MessageConverter가 Spring Web에서 동작하는 방식은 다른 포스팅에서 설명)

Jackson은 어떻게 객체를 만들까?

  • Jackson은 JSON 데이터를 Java 객체로 변환할 때 기본적으로 다음과 같은 순서로 동작합니다:
  1. 기본 생성자를 찾아 객체를 생성
  2. JSON의 각 필드값을 매핑
  3. 해당 필드에 값을 설정
JSON 데이터 예시
// 이런 JSON 데이터가 있다고 가정해봅시다
{
    "name": "김개발",
    "age": 30
}


// Jackson은 이런 클래스를 찾습니다
public class User {
    private String name;
    private int age;


    // Jackson이 필요로 하는 기본 생성자
    public User() {}


    // Getter/Setter
}

@Builder를 사용할 때 발생하는 문제

  • Lombok의 @Builder를 클래스 레벨에 적용하면 다음과 같은 변화가 일어납니다:
rtets
1@Builder
2public class User {
3private String name;
4private int age;
5

6    // Lombok이 생성하는 private 생성자
7    private User(String name, int age) {
8        this.name = name;
9        this.age = age;
10    }
11}
  • 여기서 문제가 발생합니다. Jackson이 필요로 하는 기본 생성자가 없어진 것입니다.

해결 방법과 Best Practice

  1. 생성자 레벨 @Builder 사용
public class User {
    private final String name;
    private final int age;


    @Builder
    public User(String name, int age) {
        this.name = name;
        this.age = age;
    }


    // Jackson을 위한 기본 생성자
    @JsonCreator
    public User() {
        this.name = null;
        this.age = 0;
    }
}
  1. @JsonDeserialize 활용
@JsonDeserialize(builder = User.UserBuilder.class)
@Builder
public class User {
    private final String name;
    private final int age;
}

실제 프로젝트에서의 적용

실제 프로젝트에서는 다음과 같은 패턴을 추천합니다:

  1. Request DTO의 경우
public class UserRequest {
    private final String name;
    private final int age;


    @Builder
    @JsonCreator
    public UserRequest(@JsonProperty("name") String name,
                      @JsonProperty("age") int age) {
        this.name = name;
        this.age = age;
    }
}
  1. Response DTO의 경우
@Value
@Builder
public class UserResponse {
    String name;
    int age;


    // Response는 직렬화만 필요하므로 별도의 처리가 필요 없습니다
}

성능과 관련된 고려사항

@Builder 사용 시 고려해야 할 성능 관련 포인트들:

  1. 메모리 사용량
  • Builder 클래스가 추가로 생성됨
  • 객체 생성 시 중간 객체가 생성됨
  1. 실행 시간
  • 일반 생성자 대비 약간의 오버헤드 발생
  • 대량의 객체 생성 시 영향이 커질 수 있음
// 성능 테스트 결과 예시
@Test
public void performanceTest() {
    long start = System.nanoTime();
    IntStream.range(0, 1_000_000)
        .forEach(i -> User.builder()
            .name("test")
            .age(20)
            .build());
    long builderTime = System.nanoTime() - start;


    start = System.nanoTime();
    IntStream.range(0, 1_000_000)
        .forEach(i -> new User("test", 20));
    long constructorTime = System.nanoTime() - start;


    // 결과: Builder 패턴이 약 15-20% 더 느림
}

정리: 언제 @Builder를 사용해야 할까?

  • ✅ 사용하면 좋은 경우:

    • 선택적 매개변수가 많은 경우
    • 불변 객체를 만들어야 하는 경우
    • 생성 과정에서 유효성 검사가 필요한 경우

  • ❌ 피해야 할 경우:

    • API 요청/응답을 위한 DTO 클래스 (클래스 레벨)
    • 성능이 매우 중요한 경우
    • 단순한 데이터 전달용 객체

마치며

@Builder는 분명 유용한 도구이지만, 모든 상황에 적합한 것은 아닙니다. 특히 Spring Web MVC에서 API 요청/응답을 처리할 때는 Jackson과의 상호작용을 고려해야 합니다. 이러한 이해를 바탕으로 적절한 시점에 적절한 방식으로 @Builder를 사용한다면, 더 안정적이고 유지보수하기 좋은 코드를 작성할 수 있을 것입니다.

1/7 🚨 Spring에서 자주 보는 에러 "com.fasterxml.jackson.databind.exc.InvalidDefinitionException: Cannot construct instance of ..."

왜 발생할까요? 바로 @Builder 때문입니다.

2/7 원인: Jackson은 기본 생성자가 필요한데, @Builder는 이를 제거해버립니다.
